# Research: 多集群 Kubernetes 可观测中心

## 决策 1：平台作为可观测控制面，不承担原始日志和指标存储
- Decision: 002 继续沿用 001 的控制面思路，平台自身只保存治理元数据、快照和审计信息，不把原始日志全文或原始指标点落到 MySQL。
- Rationale: 日志和时序指标的存储、压缩、保留与查询优化本身就是独立产品域。若平台自己承担原始数据存储，会显著扩大 002 的基础设施和维护复杂度，也会与“保持 001 主栈不变”的原则冲突。
- Alternatives considered: 将原始日志与指标同步到平台数据库被排除，因为会造成存储压力、同步复杂度和查询模型失真；完全前端直连外部观测后端被排除，因为无法统一授权、审计和租户隔离。

## 决策 2：指标与运行态告警查询采用 Prometheus + Alertmanager 兼容接口
- Decision: 指标查询走 Prometheus-compatible Query API，运行态告警查询与静默状态同步走 Alertmanager-compatible API，统一放在后端适配层中实现。
- Rationale: 这与用户提出的技术方向一致，也最接近 Rancher 在观测层面的生态对接模式。通过适配层统一封装后，平台可以保持查询面稳定，同时为不同兼容实现保留扩展点。
- Alternatives considered: 只接 Prometheus 被排除，因为 Prometheus 只覆盖指标与规则评估，不解决告警路由、静默与运行态告警管理；自研告警引擎被排除，因为超出 002 范围且交付风险过高。

## 决策 3：告警规则、通知目标和处理记录由平台持久化管理，再通过适配器投影到运行后端
- Decision: 告警规则、通知目标、静默窗口、处理说明和治理状态由平台保存在 MySQL，后端通过适配器把可执行部分同步到兼容的告警后端或集群侧规则对象。
- Rationale: Prometheus 和 Alertmanager 的查询接口并不能完整覆盖规则生命周期与通知目标治理。若把这些治理数据完全交给外部后端，平台无法稳定实现“创建、编辑、启用、停用、处理记录、权限和审计”这些产品能力。
- Alternatives considered: 完全把规则和接收器配置托付给外部告警系统被排除，因为平台只能做被动展示；平台只做界面，不保存任何治理元数据被排除，因为无法满足审计、权限和复盘要求。

## 决策 4：日志查询首期采用 Loki 兼容接口，平台不提供日志写入链路
- Decision: 日志查询首期通过 Loki-compatible Query API 获取，并以资源上下文、时间范围和关键字为主轴组织检索；平台本身不负责采集日志或写入日志后端。
- Rationale: 规格要求是“查看、筛选和检索日志”，不是建设日志采集系统。Loki 与 Kubernetes 场景耦合紧密，适合与资源上下文联动，且与“平台不承担原始数据存储”保持一致。
- Alternatives considered: 直接接入 Elasticsearch/OpenSearch 被保留为后续适配器扩展，而不是首期默认；首期自建日志采集链路被排除，因为超出范围。

## 决策 5：Kubernetes 事件直接通过 client-go 查询，并结合短时缓存降低重复读取
- Decision: 事件时间线数据直接来自 Kubernetes Event，平台通过 client-go 按授权范围读取，并在 Redis 中做短时缓存与去重聚合。
- Rationale: Event 是 Kubernetes 原生语义，最适合用来补充日志和指标之间的时间线信息。直接读取能最大限度保留原始语义，同时短时缓存可避免高频刷新造成集群 API 压力。
- Alternatives considered: 把所有事件长期落库被排除，因为会扩大存储范围且价值有限；完全不缓存被排除，因为列表和联动视图会产生大量重复读取。

## 决策 6：前端统一采用 ECharts，围绕资源上下文构建联动视图
- Decision: 前端继续使用 React + Ant Design 现有栈，新增 Apache ECharts 作为统一图表层，围绕资源上下文提供概览、趋势图、事件时间线和告警中心视图。
- Rationale: 002 的核心不是单页大盘，而是“从目标资源出发联动看到日志、事件、指标、告警”。ECharts 的表达能力更适合时序趋势、分层概览和告警统计等场景，也避免引入多套图表系统。
- Alternatives considered: 只用表格和数字卡片被排除，因为不利于趋势与异常变化判断；同时引入多套图表库被排除，因为会增加维护成本和视觉不一致性。

## 决策 7：查询编排采用后端扇出聚合 + 轮询刷新，不在首期引入复杂实时流
- Decision: 后端负责对 Prometheus、Alertmanager、Loki 和 Kubernetes API 进行统一扇出、聚合和错误归一化；前端首期采用轮询刷新和显式时间范围更新，不引入日志 tail 或复杂长连接流。
- Rationale: 002 首期重点是统一入口和故障定位闭环，而不是终端式实时追踪。轮询刷新能在复杂度、稳定性和交付速度之间取得更合适的平衡。
- Alternatives considered: 前端直接并行请求各观测后端被排除，因为无法统一鉴权和审计；一开始就引入 WebSocket 或 SSE 实时流被排除，因为与首期目标不匹配且复杂度过高。

## 决策 8：继续复用 001 的工作空间/项目隔离与审计模型
- Decision: 002 不重新发明权限模型，而是在现有工作空间/项目边界之上扩展观测范围判定、告警治理授权和关键观测访问审计。
- Rationale: 用户已经要求 002 必须继承现有工作空间和项目级隔离。如果在 002 中另起一套授权规则，会与 001 冲突并扩大实施风险。
- Alternatives considered: 仅依赖外部观测后端自己的租户模型被排除，因为无法与 kbManage 当前的业务边界完全一致；前端仅做入口隐藏不做后端授权被排除，因为不安全。
