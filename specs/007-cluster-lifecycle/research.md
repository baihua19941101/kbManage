# Research: 多集群 Kubernetes 集群生命周期中心

> 研究输入基于 007 规格、001-006 既有能力边界以及 Rancher 风格集群管理通用语义，目标是在不扩展到平台级灾备、统一身份源整合、策略治理和应用市场的前提下，形成可实施的首期方案。

## Decision 1: 007 采用独立 `clusterlifecycle` 业务域，而不是继续扩展 001 `cluster`

- Decision: 后端新增 `service/clusterlifecycle`、`integration/clusterlifecycle` 与对应 handler/router/repository/worker，前端新增 `features/cluster-lifecycle`；复用 001 的授权和审计底座，但生命周期动作与能力矩阵模型独立。
- Rationale: 001 更偏向“接入后资源总览与权限隔离”，007 的核心对象是“导入/注册、创建、升级、停用、退役、驱动与模板管理”。独立业务域可避免集群总览与生命周期编排职责混杂。
- Alternatives considered:
  - 继续扩展 `cluster` 服务：会把静态总览和动态生命周期动作耦合在一起，增加权限和审计语义复杂度。
  - 并入 `gitops` 或 `workloadops`: 职责不匹配，无法表达集群创建和退役等基础设施生命周期。

## Decision 2: 导入与注册分离建模，但统一归入生命周期记录

- Decision: “导入现有集群”与“注册新纳管集群”在业务上作为两类接入动作建模，分别记录来源凭据/接入信息与注册引导/确认状态，但最终都归并到统一的 `ClusterLifecycleRecord`。
- Rationale: 两类流程的前置条件、失败阶段和用户交互不同，但平台仍需要用统一视角管理后续升级、节点池和退役。
- Alternatives considered:
  - 完全统一成单一“接入”动作：难以保留不同流程的阶段语义和诊断信息。
  - 为导入和注册维护完全独立对象：会导致后续生命周期动作无法共享同一集群主记录。

## Decision 3: 创建能力采用“驱动抽象 + 模板化参数集 + 创建前校验”模式

- Decision: 平台通过驱动抽象层统一承接基础设施类型差异；模板负责固化常用参数和能力要求；创建前校验负责检查驱动版本、模板要求与目标环境是否兼容。
- Rationale: 这样既能保留不同基础设施的扩展空间，又能避免首期将平台绑定到某一个具体驱动实现。
- Alternatives considered:
  - 每种基础设施单独做一套创建流程：交互和后端模型碎片化，后续维护成本高。
  - 只做通用表单、不做模板和校验：用户需要理解过多底层差异，错误率高。

## Decision 4: 能力矩阵以“能力域 + 支持状态 + 兼容说明”建模，而非纯布尔开关

- Decision: 能力矩阵条目至少包含能力域、支持状态、兼容级别、说明文本和适用的驱动/版本范围，而不是简单记录“支持/不支持”。
- Rationale: 集群能力往往存在“原生支持、需扩展支持、部分支持、不兼容”几类状态，仅用布尔值难以支撑创建前决策和升级影响分析。
- Alternatives considered:
  - 纯布尔矩阵：表达力不足，无法说明限制条件。
  - 仅以自由文本描述：可读但不可筛选、不可校验。

## Decision 5: 驱动版本与模板版本解耦，但模板必须声明依赖驱动范围

- Decision: `ClusterDriverVersion` 与 `ClusterTemplate` 分离建模；模板需要声明适用驱动、版本范围和必需能力。驱动升级不直接改写模板，而是通过兼容性检查影响模板可用状态。
- Rationale: 驱动迭代频率与模板迭代频率不同，解耦后可以保留历史兼容性判断并支持模板逐步迁移。
- Alternatives considered:
  - 驱动与模板一一绑定：版本升级会导致模板资产频繁复制和难以维护。
  - 模板不记录驱动依赖：创建前很难做有效阻断和风险提示。

## Decision 6: 升级、节点池调整和退役采用互斥动作锁

- Decision: 对升级、节点池扩缩、停用和退役这类高影响动作采用统一的互斥锁定语义；同一集群同一时间只允许一个关键生命周期动作处于执行中。
- Rationale: 这类动作往往共享底层资源和状态判断，允许并发执行会显著增加失败和状态漂移风险。
- Alternatives considered:
  - 允许并行执行多个动作：实现复杂度高且风险大，不适合首期。
  - 完全串行化所有动作包括只读查询：限制过度，不利于运维效率。

## Decision 7: 升级计划独立建模，执行记录与集群主记录分离

- Decision: 升级计划作为独立对象记录目标版本、前置检查、风险摘要、窗口期、执行状态和结果；集群主记录仅保留当前版本、目标版本和最近升级结果摘要。
- Rationale: 升级属于可追踪的阶段性流程，需要独立查询和审计；同时又不能让集群主记录承载过多历史细节。
- Alternatives considered:
  - 将升级字段直接堆到集群主记录：历史升级链路难以复盘。
  - 把升级并入通用动作表：丢失升级特有的计划与校验语义。

## Decision 8: 节点池以结构化快照管理，而非直接映射底层节点对象

- Decision: 节点池信息以 `NodePoolProfile` 形式保存，记录角色、期望容量、当前容量、可用区/池标签、版本与状态；运行时节点明细仍通过 Kubernetes API 或驱动适配层获取。
- Rationale: 生命周期中心需要管理的是“节点池配置和状态”，而不是替代底层节点明细视图。快照模型更适合扩缩和升级计划。
- Alternatives considered:
  - 直接以节点对象集合表达节点池：难以管理池级操作和配置意图。
  - 只保存节点池名称不保留容量状态：无法满足扩缩与升级场景。

## Decision 9: 退役流程采用“停用 -> 校验 -> 退役完成”两阶段模型

- Decision: 首期退役流程分为停用和退役两个阶段；停用用于阻止新动作和提示迁移，退役完成用于记录资源释放与最终结论。若存在阻断项，退役必须停留在中间状态。
- Rationale: 直接从活跃状态切到已退役会掩盖迁移与清理过程，也不利于审计复盘。
- Alternatives considered:
  - 单步退役：流程简单但风险提示不足。
  - 增加过多中间阶段：首期实施成本和用户理解成本过高。

## Decision 10: 审计查询复用独立生命周期动作域，而不混入通用资源动作

- Decision: 007 的审计事件单独归类为集群生命周期动作域，覆盖导入、注册、创建、校验、升级、节点池调整、停用、退役、驱动变更和模板变更。
- Rationale: 生命周期动作对象大多以“集群/驱动/模板”为中心，与 003 的资源运维动作和 004 的发布动作不同，单独域更利于筛选和复盘。
- Alternatives considered:
  - 混入通用审计动作集合：查询维度模糊，不利于职责区分。
  - 只保留粗粒度动作：无法满足关键动作审计要求。
